## TL;DR

Claude Code を放置していたら **メモリが 120GB 超**まで膨らみ、Mac が実質フリーズしました。
再起動以外に復旧手段がなく、過去に 3 回同じ目に遭っています。3 回とも Ghostty + Claude Code の組み合わせで、放置時間は最短 1 時間でした。

対策として、**RSS が閾値を超えた Claude プロセスを自動 kill する watchdog** を `launchd` で常駐させています。導入以降フリーズは起きておらず、実際に **RSS 9.5GB まで膨らんだプロセスを OS がフリーズする前に自動停止できた**実績があります。

この記事では、症状の見分け方から watchdog の導入手順までをまとめます。

## 何が起きるのか

長時間放置した Claude Code セッションで、以下のような状態になることがあります。

1. **ターミナルが無反応になる** — Ctrl+C も効かない
2. **アクティビティモニタで Claude プロセスの RSS が異常に大きい** — 手元では 120GB 超
3. **スワップが物理メモリを大幅に超え、macOS 全体がカクカクになる**
4. **プロセスを kill しても OS の状態が戻らず、再起動が必要になる**

| 項目        | 値                             |
| ----------- | ------------------------------ |
| マシン      | MacBook Pro（M4 Pro / 48GB）   |
| ターミナル  | Ghostty 1.3.1                  |
| Claude Code | 2.1.78 / 2.1.80                |
| 放置時間    | 1〜4 時間（最短 1 時間で発生） |

## 原因はどこにあるのか

正直に言うと、**単一の原因を断定できていません**。ただし切り分けの材料はあります。

### Ghostty 側

Ghostty には、大量出力を契機にメモリが膨らむバグの修正履歴があります。

- [Ghostty 1.3.0 リリースノート](https://ghostty.org/docs/install/release-notes/1-3-0)
- [mitchellh によるメモリリーク修正の解説](https://mitchellh.com/writing/ghostty-memory-leak-fix)

ただし手元は修正済みの 1.3.1 以降で発生しており、Ghostty 単体では説明しきれません。

### Claude Code 側

コミュニティでも Claude Code のメモリ増加は複数報告されています。

- [Reddit: Claude Code のメモリリーク PSA](https://www.reddit.com/r/ClaudeAI/comments/1rr65dz/psa_claude_code_v2172v2173_has_confirmed_memory/)
- [Reddit: idle 時に 80MB/min ペースで増加する報告](https://www.reddit.com/r/ClaudeCode/comments/1rrdtvt/claude_code_eats_80_mbmin_of_ram_sitting_idle/)

手元でも **Claude プロセス自体の RSS が大きかった**ため、こちらの比重が大きいと考えています。

### ステータスライン

自作のステータスラインを使っていましたが、スクリプトに明らかなリークは見当たりませんでした。ただしコミュニティではステータスライン周辺が悪化要因になりうるという話もあり、外部コマンドの呼び出し頻度には注意が必要です（[公式ドキュメント](https://code.claude.com/docs/en/statusline)）。

## 症状が出たときの初動

フリーズしかけの状態で慌てないために、やることを整理しておきます。

1. **別のターミナルかアクティビティモニタを開く**（メインのターミナルは操作不能の前提）
2. **アクティビティモニタでメモリ順にソート**し、どのプロセスが肥大化しているか確認する
3. **該当プロセスを kill する**（`kill -15` → 5 秒待って `kill -9`）
4. **それでも OS が重いなら再起動する** — スワップが大量に残っていると kill だけでは戻らない

## 対策: watchdog で自動 kill する

毎回手動で対応するのは現実的ではないので、**RSS が閾値を超えた時点で自動的にプロセスを止める watchdog** を作りました。

### 考え方

- 通常時の Claude プロセスは **おおよそ 200MB 前後**
- **4GB を超えたら異常**とみなす — 通常 200MB に対して十分大きく、かつ物理メモリ（48GB）を食い潰す前に止められるライン
- `launchd` で **30 秒ごと**にチェック
- `SIGTERM` → 5 秒後に生存確認 → `SIGKILL` の 2 段階

閾値はマシンのメモリ量に応じて調整してください。動作確認は閾値を 100MB 程度に下げて Claude Code を起動すれば、すぐに kill されることで検証できます。

### スクリプト

配置場所は任意です。ここでは `~/.local/share/claude-mem-watchdog/` を使います。

```bash:kill-claude-on-high-mem.sh
#!/bin/bash
# Claude プロセスの RSS が閾値を超えたら終了させる（macOS / bash）

THRESHOLD_KB=$((4 * 1024 * 1024)) # 4GiB を KB 換算（ps の rss は KB）
LOG_DIR="$HOME/.local/share/claude-mem-watchdog"
DEBUG_LOG="$LOG_DIR/mem-watchdog.log"
EVENT_LOG="$LOG_DIR/mem-watchdog-events.log"

mkdir -p "$LOG_DIR"
ts() { date '+%Y-%m-%d %H:%M:%S'; }

ps -eo pid,rss,command | grep -i '[c]laude' | while read -r pid rss cmd; do
  [ -z "$rss" ] && continue
  rss_mb=$((rss / 1024))
  threshold_mb=$((THRESHOLD_KB / 1024))

  if [ "$rss" -gt "$THRESHOLD_KB" ] 2>/dev/null; then
    msg="[$(ts)] KILL PID=$pid RSS=${rss_mb}MB (>${threshold_mb}MB) CMD=$cmd"
    echo "$msg" >> "$DEBUG_LOG"
    echo "$msg" >> "$EVENT_LOG"
    logger -t claude-mem-watchdog "Killing Claude (PID=$pid) using ${rss_mb}MB (>${threshold_mb}MB)"
    kill -15 "$pid"
    sleep 5
    if kill -0 "$pid" 2>/dev/null; then
      kill -9 "$pid" 2>/dev/null
      msg="[$(ts)] SIGKILL sent to PID=$pid"
      echo "$msg" >> "$DEBUG_LOG"
      echo "$msg" >> "$EVENT_LOG"
    fi
  else
    echo "[$(ts)] OK   PID=$pid RSS=${rss_mb}MB (limit=${threshold_mb}MB)" >> "$DEBUG_LOG"
  fi
done
```

### launchd の設定

```bash
# ディレクトリ作成とスクリプトに実行権限を付与
mkdir -p ~/.local/share/claude-mem-watchdog
chmod +x ~/.local/share/claude-mem-watchdog/kill-claude-on-high-mem.sh
```

以下の plist を `~/Library/LaunchAgents/com.user.claude-mem-watchdog.plist` に保存します。
**`YOUR_NAME` は自分のユーザー名に置き換えてください**（plist 内では `$HOME` は展開されません）。

```xml:com.user.claude-mem-watchdog.plist
<?xml version="1.0" encoding="UTF-8"?>
<!DOCTYPE plist PUBLIC "-//Apple//DTD PLIST 1.0//EN"
  "http://www.apple.com/DTDs/PropertyList-1.0.dtd">
<plist version="1.0">
<dict>
  <key>Label</key>
  <string>com.user.claude-mem-watchdog</string>
  <key>ProgramArguments</key>
  <array>
    <string>/bin/bash</string>
    <string>/Users/YOUR_NAME/.local/share/claude-mem-watchdog/kill-claude-on-high-mem.sh</string>
  </array>
  <key>StartInterval</key>
  <integer>30</integer>
  <key>RunAtLoad</key>
  <true/>
  <key>StandardOutPath</key>
  <string>/Users/YOUR_NAME/.local/share/claude-mem-watchdog/mem-watchdog-launchd.log</string>
  <key>StandardErrorPath</key>
  <string>/Users/YOUR_NAME/.local/share/claude-mem-watchdog/mem-watchdog-launchd.log</string>
</dict>
</plist>
```

```bash
# 読み込み・確認
launchctl load ~/Library/LaunchAgents/com.user.claude-mem-watchdog.plist
launchctl list | grep claude-mem

# ログを確認
tail -f ~/.local/share/claude-mem-watchdog/mem-watchdog.log
```

```bash
# 停止したいとき
launchctl unload ~/Library/LaunchAgents/com.user.claude-mem-watchdog.plist
```

## 運用上の注意

- kill は「悪化の歯止め」であり、すでに膨らんだスワップを解消するわけではありません。kill 後も OS が重い場合は再起動してください。
- `grep '[c]laude'` はコマンド行に `claude` を含むプロセスすべてにマッチします。名前が似た別プロセスを誤 kill する可能性はゼロではありません。
- セッションが突然終了するため、`claude --resume` や `claude --continue` での復帰が前提の運用になります。
- 導入直後は `mem-watchdog.log` を眺めて、意図しないマッチや過剰な kill がないか確認してください。

## 追記1: watchdog が実際に発動した（2026-03-20）

記事公開後、**watchdog が実際にメモリ暴走を検知・停止しました**。

```
[2026-03-20 19:20:44] OK   PID=58286 RSS=233MB (limit=4096MB)
[2026-03-20 19:20:44] OK   PID=58288 RSS=112MB (limit=4096MB)
[2026-03-20 19:20:44] KILL PID=47011 RSS=9741MB (>4096MB) CMD=claude --dangerously-skip-permissions
[2026-03-20 19:20:49] SIGKILL sent to PID=47011
```

RSS が **約 9.5GB** まで膨らんだ Claude プロセスを検知し、`SIGTERM` → 5 秒後に `SIGKILL` の 2 段階で停止しています。4GB の閾値で十分早く捕捉でき、OS がフリーズする前に食い止められました。

以前は「お守り代わり」と書いていましたが、**実際に機能することが確認できた**ので、導入しておく価値はあると言えます。

## 追記2: 監視間隔を 60 秒 → 30 秒に変更した（2026-03-20）

上記のログを見ると、前回の OK チェックと KILL チェックが**同じ 19:20:44 のタイミング**に記録されています。つまり、少しずつ増えるのではなく、**60 秒以内に一気に数GB単位で膨らんでいる**ことがわかりました。

60 秒間隔だと気づいたときにはすでに大量消費している可能性があるため、`StartInterval` を 30 秒に変更しました。この記事に掲載しているスクリプトも 30 秒に修正しています。

```xml
<key>StartInterval</key>
<integer>30</integer>
```

plist を変更後、以下で再読み込みします。

```bash
launchctl unload ~/Library/LaunchAgents/com.user.claude-mem-watchdog.plist
launchctl load  ~/Library/LaunchAgents/com.user.claude-mem-watchdog.plist
```

## 補足: 放置しない運用も有効

watchdog とは別に、**長時間使わないときは Claude Code を終了し、再開時に `--resume` / `--continue` で戻る**運用をしています。そもそもプロセスを放置しなければメモリが膨らむ機会がないため、これだけでもリスクはかなり下がります。

ただし上記の追記のとおり、放置しない運用を心がけていても再発しました。**放置しない + 万が一のために watchdog を入れておく**、という二段構えが現時点での着地点です。

再発すると OS 再起動が必要になり、ウィンドウ配置の復元など地味に手間がかかるので、常駐させ続けることをおすすめします。
